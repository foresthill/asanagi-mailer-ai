# 設計書 02 — 重要度の仕分け画面と、ユーザーごとの学習機構

最終更新: 2026-06-03

## 1. やりたいこと

1. **仕分け画面**: 重要メールの一覧。横カラムに「**なぜ重要か（理由）**」。
2. **フィードバック**: AIの推測が合っているか（👍/👎）。**押さなくてもいい**が、押すと学習に溜まる。
3. **per-user 最適化**: デフォルトのAI判定に加え、その人専用に賢くなる。

本書は「**そのための最適な学習機構**」をエンジニアリング判断として提案する。

---

## 2. 結論（推奨アーキテクチャ）

> **ファインチューニングはしない。** 「決定的シグナル + 埋め込みRAG(k-NN) + 嗜好プロファイル + （後に）軽量分類器」の**4層ハイブリッド**を推奨する。

理由は §4 の比較で詳述。まず全体像:

```
入力メール
   │
   ▼
┌─ 層0: 決定的シグナル ──────────────┐  即時・無料・説明可能
│  送信者/ドメイン/List-Id ルール     │  「Acme社は常に重要」等を即断
└───────────────┬────────────────────┘
                │ ヒットすれば早期確定（LLM呼ばない）
                ▼ ミスなら
┌─ 層1: 埋め込みRAG (k-NN) ──────────┐  これが「RAG」の実体
│  メールを埋め込み→過去の            │  8万通でもベクトル検索でスケール
│  「判定済み類似メール」をtop-k取得   │
└───────────────┬────────────────────┘
                ▼
┌─ 層2: LLM最終判定 ─────────────────┐  文脈で判断・理由生成
│  入力 ＋ 取得した類似例(few-shot)    │  importance + reason + confidence
│  ＋ 嗜好プロファイル(自然言語)       │
└───────────────┬────────────────────┘
                ▼
        重要度 + 理由 + 確信度
                │
   フィードバック(👍/👎/暗黙) を学習ストアへ
                │
   ┌─ 層3(後日): 軽量分類器 ──────────┐  ラベルが貯まったら
   │  埋め込み+特徴量でロジスティック   │  端末ローカルで秒で学習
   │  回帰/GBDT。較正された確率を       │  LLMの強い事前確率に
   └────────────────────────────────────┘
```

## 3. 各層の詳細

### 層0 — 決定的シグナル（実装済みの発展）
送信者・ドメイン・`List-Id`（メーリングリスト）・キーワードに重み付きラベルを持つ。
- 「この送信者は常に重要/常に低」をユーザーが明示すれば**即断**（LLM不要＝速い・安い・完全に説明可能）。
- 現状の `src/lib/store.ts` の `ImportanceSignal` を拡張（`list-id`, `confidence減衰` を追加）。
- 実際のメールの相当割合（通知・請求・特定取引先）はここで確定できる。

### 層1 — 埋め込みRAG（k-NN）＝ ユーザーの言う「RAG」
- 各メールを**埋め込みベクトル**化（subject + 本文先頭 + 送信者）。
- 判定済みメールを `(vector, label, reason)` で**ベクトルストア**に保存。
- 判定時、入力メールに**最も似た過去の判定済みメールを top-k 取得**。
- 使い方は2通り（併用）:
  - **(a) 重み付き k-NN 投票**: 距離で重み付けした多数決 → 高速な事前確率。
  - **(b) few-shot 文脈**: 取得例をLLMプロンプトへ「過去の類似判定」として注入。
- **なぜRAGか**: 8万通の判定履歴を全部プロンプトに入れるのは不可能。ベクトル検索なら「**今のメールに関係する数件だけ**」を引ける＝スケールする。これが「RAGとして溜まっていく」の技術的実体。

### 層2 — LLM最終判定
入力 = メール本文 ＋ 層0シグナル ＋ 層1で取得した類似例 ＋ **嗜好プロファイル**。
出力 = `importance(high/normal/low)` ＋ `reason`（日本語1〜2文）＋ `confidence(0–1)`。
- confidence が低い時だけユーザーに確認を促す（能動学習＝少ない操作で賢くなる）。

### 層3 — 軽量分類器（ラベルが十分貯まってから / Phase 2〜3）
- ラベルが例えば200件超で、埋め込み＋メタ特徴量（送信者頻度・件名に疑問符・スレッド深さ・宛先がToかCcか・時間帯・既読までの時間 等）から**ロジスティック回帰 / GBDT**を学習。
- 数秒で再学習でき、**端末ローカルで完結**。較正された確率を返し、LLMの強い事前確率として使う（最終文・理由はLLM）。
- ここで初めて「その人だけのモデル」が育つ。RAGと併存。

## 4. なぜこの構成か（手法比較）

| 手法 | 初期精度 | 学習遅延 | 必要データ量 | オンライン更新 | 説明可能性 | コスト | プライバシー(ローカル可) |
|------|---------|---------|-------------|--------------|-----------|-------|----------------------|
| ルール/シグナル(層0) | 中 | 即時 | 少 | ◎ | ◎ | 無料 | ◎ |
| Few-shot(プロンプト) | 中〜高 | 即時 | 少 | ◎ | ○ | LLM呼出 | △ |
| **埋め込みRAG/k-NN(層1)** | 高 | 即時 | 中 | ◎ | ○(類似例提示) | 埋め込み＋検索 | ◎ |
| 軽量分類器(層3) | 高 | 秒 | 中〜多 | ○(再学習) | ○(係数) | 極小 | ◎ |
| **ファインチューニング** | 高 | 時間〜日 | 多(千〜) | ✗(再学習重い) | ✗ | 高 | ✗(学習基盤依存) |

**ファインチューニングを採らない理由**:
- per-user に毎回ファインチューニングするのは**コスト・運用が破綻**する（ユーザー数ぶんのモデル管理）。
- フィードバックが反映されるまで**遅い**（即時性が命のメール体験に致命的）。
- **大量のラベル**が必要だが、メールの重要度ラベルはユーザーが少しずつしか与えない。
- **説明できない**（「なぜ重要と判定したか」が出せない＝透明性原則に反する）。
- 会社メールでは学習基盤にデータを出せない（**プライバシー違反**）。

→ RAG＋小分類器なら、**即時・安い・説明可能・ローカル完結**で、フィードバックが貯まるほど確実に賢くなる。これが本用途の最適解。

## 5. フィードバックの設計（👍/👎 を「学習」にどう変えるか）

### 5.1 何を記録するか
毎回の判定をイベントとして残す:
```ts
interface ClassificationEvent {
  emailId: string;
  features: FeatureVector;       // 送信者頻度・件名特徴・スレッド深さ 等
  embedding: number[];           // 層1用
  aiPrediction: Importance;
  aiConfidence: number;
  userVerdict?: Importance;      // 👍=aiPredictionと同じ / 👎=訂正後ラベル
  source: "explicit" | "implicit";
  timestamp: string;
}
```

### 5.2 ボタンの意味づけ（ここが肝）
- **👍（合ってる）** → ラベル = AI予測。シグナル＆ベクトルストアを**強化**。
- **👎（違う）** → **訂正後の正解ラベルを取る**（👎だけでは「何が正解か」が分からない）。UIは👎を押すと「では正しくは？ 重要/通常/低」を即提示。**訂正データは最も価値が高い教師信号**。
- **押さない** → 「正解」と見なさない（unlabeled のまま）。誤った自己強化を避ける。

### 5.3 暗黙シグナル（ボタンを押さなくても学習）★重要
ユーザーの自然な操作も弱いラベルとして使う:
| 行動 | 含意 |
|------|------|
| 開封してすぐ返信 | 重要だった可能性 高 |
| 開かずアーカイブ/ゴミ箱 | 低だった可能性 高 |
| スターを付けた | 重要 |
| 長く滞在して読んだ | 重要寄り |

暗黙シグナルは弱い重み、明示フィードバック（👍/👎）は強い重みで学習に反映。これで「**押さなくても勝手に賢くなる**」を実現する。

### 5.4 嗜好プロファイルの蒸留（learned instructions）
N件の新フィードバックごとに、LLMで履歴を要約し**自然言語の嗜好プロファイル**を更新:
> 例: 「重要: 取引先Acmeの返信要求、請求/契約、上司Tanakaからの直接依頼。低: ニュースレター、SaaSの自動通知、CC止まりの連絡。」

これを毎回の判定プロンプトに注入。**安い・説明可能・即効**で、RAGと相補的。そして次節のとおり**ユーザーが直接編集できる**。

## 6. UI設計

### 6.1 仕分け画面（Triage）
```
┌─ 仕分け ──────────────────────────────────────────────┐
│  [重要 3] [通常 8] [低 12]            確信度低のみ表示 ▢ │
├────────────────────────────────────────────────────────┤
│ ● 佐藤(Acme)  来週の打合せ日程   │ 取引先からの返信要求 │ 👍 👎 │
│   件名/送信者/プレビュー          │ ＝AIの理由（横カラム）│ 合/違 │
│ ● billing      5月請求のお知らせ  │ 請求・期限あり        │ 👍 👎 │
│ ○ TechLetter   今週の注目10選     │ ニュースレター(自動)  │ 👍 👎 │
└────────────────────────────────────────────────────────┘
```
- 行 = メール、横カラム = **AIの理由**、右 = **👍/👎**（押さなくてよい）。
- 👎を押すと正解ラベルの選択がインライン展開。
- （発展）重要度カラムへの**ドラッグで訂正**も強い教師信号にする（kanban風）。

### 6.2 学習ビュー（per-user最適化を「見える化」）
- **嗜好プロファイル**を表示し、**ユーザーが直接編集可能**（明示・暗黙・手動編集の3経路）。
- 学習済みシグナル（送信者/ドメイン）の一覧・無効化・削除。
- 精度の推移（フィードバックでどれだけ改善したか）。

> ブラックボックスにしない ＝ 透明性原則。ユーザーが「自分のAI」を理解・調整できることが per-user 最適化のUX的な核心。

## 7. ストレージ／技術選定

| フェーズ | 埋め込み | ベクトル検索 | 学習ストア |
|----------|---------|-------------|-----------|
| Phase 1 (Web) | プロバイダAPI（例: OpenAI `text-embedding-3-small` / Voyage）。`AI_PROVIDER`同様に抽象化 | 件数小ならJSでcosine。増えたら **SQLite + `sqlite-vec`** | SQLite |
| Phase 2 (会社/ローカル/Mac) | **ローカル埋め込み**（端末内モデル）でメールを外部に出さない | ローカル `sqlite-vec` | ローカルSQLite |

埋め込みも `resolveModel()` 同様にプロバイダ抽象化し、Phase2でローカルへ差し替え可能にする。

## 8. 段階導入

1. **Phase 1**: 層0（シグナル, 実装済）＋ 層2（LLM判定＋理由, 実装済）＋ 仕分け画面＋👍/👎＋嗜好プロファイル。
2. **Phase 1.5**: 層1（埋め込みRAG）と暗黙シグナル。
3. **Phase 2〜3**: 層3（軽量分類器）＋ローカル埋め込み＋学習ビューの精度可視化。

## 9. 受け入れ基準

- [ ] 仕分け画面で「メール／理由／👍👎」が一覧表示される
- [ ] 👎で正解ラベルを取得し、訂正が次回判定に反映される
- [ ] 👍/👎を押さなくても、アーカイブ等の暗黙行動が弱ラベルとして蓄積される
- [ ] 嗜好プロファイルが自動更新され、かつユーザーが直接編集できる
- [ ] 類似の過去判定が判定理由に現れる（「過去に似たメールXを重要と判定」）
- [ ] 8万通規模でも判定が実用速度（ベクトル検索でtop-kのみ参照）
