# 設計書 04 — スレッド表示とインボックスの整理（Phase 2）

最終更新: 2026-06-04

## 0. 背景（課題）

> 「今のMacのメーラーは、送信者などでメールが散らばって無茶苦茶見づらい。これを綺麗に整理したい。」

Asanagi の核は **“受信箱が澄む”**。そのためには「1通ずつバラバラ」ではなく、
1. **会話を1つにまとめる（スレッド表示）**
2. **似たものを束ねて一望できる（整理＝バンドル／ビュー）**

の2軸が要る。本書はこの2軸を設計する。**Phase 2** で実装。

---

## 1. スレッド表示（会話をひとまとめに）

### 1.1 体験
- 受信箱の一覧では、同じ会話は**1行に集約**（件数バッジ「3」）。最新の差出人・要旨を代表表示。
- 開くと**会話スタック**（古い順 or 新しい順）。各メッセージは折りたたみ、最新だけ展開。
- 返信は会話の文脈（直近メッセージ）を持って [01](01-incremental-reply-editing.md) の添削に入る。
- （AI活用）長いスレッドは**「これまでの要約」**をAIが先頭に提示（例: 「日程はX→Yに変更、未決は会場」）。

### 1.2 スレッドの組み立て方
| プロバイダ | 方法 |
|-----------|------|
| **Gmail** | ネイティブの `threadId` をそのまま使う（最も正確） |
| **IMAP** | サーバが `THREAD` 拡張対応なら `THREAD=REFERENCES`。非対応ならクライアント側で `References` / `In-Reply-To` ヘッダから **JWZ アルゴリズム**で再構成 |

### 1.3 データモデル
既存 `Email` は `threadId` を保持済み。表示用に集約ビューを追加:
```ts
interface Thread {
  id: string;              // threadId
  subject: string;         // 代表（通常は最古の件名から "Re:" を除去）
  participants: EmailAddress[];
  messages: Email[];       // 時系列
  lastDate: string;
  unread: boolean;
  importance?: Importance; // スレッド単位の集約（最大値など）
  messageCount: number;
}
```
- `EmailProvider` に `listThreads(state)` を追加（Gmailは `threadId` でgroup、IMAPは上記方式）。既存 `list()` は維持。

### 1.4 操作
- アーカイブ/ゴミ箱は**スレッド単位**を既定（会話ごとさばく＝消し込みが速い）。個別メッセージ単位も可。
- 「スレッド表示 ⇄ 個別表示」をトグルで切替可能に。

**出典（スレッド）**:
- [RFC 5256 — IMAP SORT and THREAD（ORDEREDSUBJECT / REFERENCES）](https://datatracker.ietf.org/doc/html/rfc5256)
- [JWZ message threading algorithm](https://www.jwz.org/doc/threading.html)
- Gmail: `threadId`（[users.threads](https://developers.google.com/workspace/gmail/api/reference/rest/v1/users.threads)）

## 2. インボックスの整理（バンドル／ビュー）

「送信者で散らばって見づらい」を解く。鍵は **AIによる自動カテゴリ分け** ＋ **束ねて一望できるUI**。

### 2.1 カテゴリ（重要度とは別軸）
[02](02-importance-triage-learning.md) の判定に、**種別（category）**を追加する:

| category | 例 | 既定の扱い |
|----------|----|-----------|
| `person` | 人からの個別メール | 最優先・展開 |
| `notification` | GitHub等の自動通知 | 束ねる |
| `newsletter` | 配信・メルマガ | 束ねる・既読化しやすく |
| `receipt` | 請求・領収・予約確認 | 束ねる（後で参照） |
| `calendar` | 招待・日程 | 束ねる＋日付で |
| `promotion` | 宣伝 | 束ねる・低 |

- category も**学習対象**（[02](02-importance-triage-learning.md) と同じ機構: シグナル＋RAG＋嗜好プロファイル）。送信者/ドメインで安定するので層0で大半が即断できる。

### 2.2 整理ビュー（3つの見せ方を切替）
1. **バンドル表示（既定）**: カテゴリで束ねて折りたたみ。
   ```
   ▸ 👤 人から (3)            ← 展開して個別に見る
   ▸ 🔔 通知 (12)            ← まとめて既読/アーカイブ
   ▸ 🧾 請求・領収 (2)
   ▸ 📰 ニュースレター (8)    ← 一括でさばける
   ```
   束ねた単位で**「全部アーカイブ」「全部既読」**＝消し込みが一気に進む。
2. **送信者レーン表示**: 送信者/ドメインごとにグルーピング。ノイズの多い送信者を畳んで、人を上に。
3. **スマートビュー（保存フィルタ）**: 「重要のみ」「人から」「未読」「今日」など。サイドバーに常駐。

### 2.3 時間グルーピング
どのビューでも一覧は **今日 / 今週 / それ以前** のセクションに（走査しやすさ）。

### 2.4 「澄ます」操作（消し込みの加速）
- バンドル右の **一括操作**（全アーカイブ/全既読/全ゴミ箱）。
- 「ニュースレターは常に束ねて自動既読」のような**ルール化**（カテゴリ学習と連動）。
- 目標: 朝、`person`/`重要` だけ数件残し、残りは束ねて一掃 → **受信箱が澄む**。

## 3. アーキテクチャ整合

- `EmailProvider` に `listThreads(state)` を追加（既存 `list()` と併存）。
- 分類は [02](02-importance-triage-learning.md) の機構を **(importance, category) 同時出力**に拡張。`Email` に `category?: Category` を追加。
- バンドル/ビューは**クライアント側の集約**＋サーバの分類結果で構成。ローカルファースト（[03](03-providers-sync-native.md)）のSQLiteを正本に、スレッド/カテゴリのインデックスを持つ。

## 4. 受け入れ基準

- [ ] 同一会話が一覧で1行に集約され、開くと会話スタックで読める（Gmail threadId / IMAP References）
- [ ] スレッド単位でアーカイブ/ゴミ箱できる
- [ ] メールが category 別バンドルに自動整理され、束ねて一括処理できる
- [ ] 送信者レーン表示／スマートビューに切替できる
- [ ] 一覧が 今日/今週/それ以前 で区切られる
- [ ] category も誤りを学習して改善する

## 5. オープンな決定事項（要相談）
1. 既定ビューは「バンドル（カテゴリ）」か「スレッドのみ」か。
2. 自動既読/自動アーカイブのルール化をどこまで踏み込むか（攻めると見落としリスク）。
3. スレッド要約をデフォルトONにするか（AIコスト）。
