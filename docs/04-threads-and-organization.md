# 設計書 04 — スレッド表示とインボックスの整理（Phase 2）

最終更新: 2026-06-12（§1.5 実装メモ・§1.6 一覧のスレッド集約・§2.5 スター付きビューを追記）

## 0. 背景（課題）

> 「今のMacのメーラーは、送信者などでメールが散らばって無茶苦茶見づらい。これを綺麗に整理したい。」

Asanagi の核は **“受信箱が澄む”**。そのためには「1通ずつバラバラ」ではなく、
1. **会話を1つにまとめる（スレッド表示）**
2. **似たものを束ねて一望できる（整理＝バンドル／ビュー）**

の2軸が要る。本書はこの2軸を設計する。**Phase 2** で実装。

## 1. スレッド表示（会話をひとまとめに）

### 1.1 体験
- 受信箱の一覧では、同じ会話は**1行に集約**（件数バッジ「3」）。最新の差出人・要旨を代表表示。
- 開くと**会話スタック**（古い順 or 新しい順）。各メッセージは折りたたみ、最新だけ展開。
- 返信は会話の文脈（直近メッセージ）を持って [01](01-incremental-reply-editing.md) の添削に入る。
- （AI活用）長いスレッドは**「これまでの要約」**をAIが先頭に提示（例: 「日程はX→Yに変更、未決は会場」）。

### 1.2 スレッドの組み立て方
| プロバイダ | 方法 |
|-----------|------|
| **Gmail** | ネイティブの `threadId` をそのまま使う（最も正確） |
| **IMAP** | サーバが `THREAD` 拡張対応なら `THREAD=REFERENCES`。非対応ならクライアント側で `References` / `In-Reply-To` ヘッダから **JWZ アルゴリズム**で再構成 |

### 1.3 データモデル
既存 `Email` は `threadId` を保持済み。表示用に集約ビューを追加:
```ts
interface Thread {
  id: string;              // threadId
  subject: string;         // 代表（通常は最古の件名から "Re:" を除去）
  participants: EmailAddress[];
  messages: Email[];       // 時系列
  lastDate: string;
  unread: boolean;
  importance?: Importance; // スレッド単位の集約（最大値など）
  messageCount: number;
}
```
- `EmailProvider` に `listThreads(state)` を追加（Gmailは `threadId` でgroup、IMAPは上記方式）。既存 `list()` は維持。

### 1.4 操作
- アーカイブ/ゴミ箱は**スレッド単位**を既定（会話ごとさばく＝消し込みが速い）。個別メッセージ単位も可。
- 「スレッド表示 ⇄ 個別表示」をトグルで切替可能に。

**出典（スレッド）**:
- [RFC 5256 — IMAP SORT and THREAD（ORDEREDSUBJECT / REFERENCES）](https://datatracker.ietf.org/doc/html/rfc5256)
- [JWZ message threading algorithm](https://www.jwz.org/doc/threading.html)
- Gmail: `threadId`（[users.threads](https://developers.google.com/workspace/gmail/api/reference/rest/v1/users.threads)）

### 1.5 実装メモ（2026-06-12 時点の現実装）

設計との差分・実装で確定した事項。コードの正は `lib/email/gmail.ts` / `lib/email/imap.ts` / `app/api/threads/[id]/route.ts`。

**スレッドの組み立て（読み取り側）**
- 一覧の集約は §1.6（クライアント側集約、2026-06-12実装）。メールを開くと `/api/threads/{account}/{threadId}` で会話を取得し、本文エリアに **ThreadView**（カード形式⇄LINE風バブル形式のトグル、選択は localStorage 永続）で表示する。
- **Gmail**: ネイティブ `threadId` ＋ `users.threads.get` のサーバサイドスレッド（設計どおり）。
- **IMAP**: `threadId` = **References チェーンの先頭（会話のルート Message-ID）** ?? In-Reply-To ?? 自身の Message-ID。mailparser で References を解析し、山括弧の有無を正規化して比較する。フルJWZ（件名フォールバック・孤児結合）は未実装の簡易版だが、準拠クライアント同士なら任意の深さで同一ルートに解決される。会話の取得はローカルキャッシュ横断（受信箱＋送信箱＋アーカイブ）。

**スレッドへの参加（送信側）— ここが落とし穴だった**
- **Gmail はヘッダだけではスレッド化されない**。`messages.send` のリクエストに `threadId` を明示し、かつ In-Reply-To/References ヘッダと件名一致（Re: 維持）の3点が揃って初めて既存スレッドに入る（[公式](https://developers.google.com/workspace/gmail/api/guides/threads)）。`OutgoingMessage.threadId` で返信元の threadId を送信リクエストまで貫通させている。
- **IMAP** 送信時は References の**先頭に会話ルートの Message-ID** を置く（`<root> <parent>` の順）。これでどの深さの返信も同じルートに解決される。threadId が uid フォールバック（`@` を含まない）の場合は References に入れない。
- **転送（Fwd:）は新しい会話**として扱う（In-Reply-To/threadId を付けない）仕様。

**関連機能**
- 返信済みマーカー（↩）はヘッダではなく、ローカルキャッシュ上「同一 thread_id に自分の sent がある」判定（`repliedThreadIds`）。送信直後に `after()` で送信箱キャッシュを更新するため、返信した瞬間からスレッド・マーカー両方に反映される。

### 1.6 一覧のスレッド集約 — 調査と設計（2026-06-12 実装）

「一覧でスレッドであることが分かるべき」に対する方式調査と採用設計。コードの正は `components/mail/threadList.ts` / `EmailList.tsx` / `MailApp.tsx`。

**方式調査（2026-06時点の観察に基づく設計判断）**

| 方式 | 代表例 | 評価 |
|------|--------|------|
| ① ツリー/階層展開 | 旧Thunderbird | 一覧内にインデントで子メールがぶら下がる。情報量最大だが行高が不揃いになり、j/kナビと相性が悪く、「受信箱を澄ませる」と逆方向。新興メーラーでの採用例はほぼ消滅 |
| ② 行内チェブロン展開 | Apple Mail / Outlook | 1行に畳み `>` で行内展開。展開状態の管理が複雑な割に「開けば見られる」ので効果薄 |
| ③ **フラット集約（採用）** | Gmail / Superhuman / Missive | 同じ会話は**常に1行**。件数バッジ＋参加者連名で「束なっている」ことを行内表現 |

採用理由: **一覧の仕事は「どの会話をさばくか選ぶこと」であって会話を読むことではない**。中身は開いた後の ThreadView（カード/LINE風）が既に担っている。キーボード速度重視の新興メーラーが③に収束しているのも同じ理屈。
参考: [Gmail conversation view](https://www.googally.com/blog/gmail-conversation-view) / [Superhuman: email threads](https://blog.superhuman.com/email-thread/) / [Missive: email thread](https://missiveapp.com/blog/email-thread) / [Gmail API: threads](https://developers.google.com/gmail/api/guides/threads)

**採用設計**

```
● 佐藤 みなみ、自分 ③        ↩ 2時間前
  Re: キックオフ日程
  承知しました。では水曜15時で…   ← 最新メッセージの要旨
```

- **クライアント側集約**（`buildRows`）: 取得済み一覧を `{account}|{threadId}` でグルーピング。代表＝最新メッセージ（snippet/日時/重要度チップの出所）。
- **行内表現**: 件数バッジ（2通以上のみ）＋**参加者連名**（古い順・自分の送信は「自分」表記・3人超は「他N人」）＋会話内に未読があれば未読扱い（太字・青ドット）＋メンバーにスターがあれば★。
- **操作は会話単位**: 行のアーカイブ/ゴミ箱・キーボード E/# は**会話の全メンバー**に作用（`ThreadRow.ids`）。トーストに「（会話N通）」と明示。
- **トグル**: ヘッダのレイヤーアイコンで「スレッド表示⇄個別表示」（localStorage 永続・既定ON）。Gmailで唯一不評な「束ねを解けない」問題への逃げ道。
- **検索結果は常に個別表示**: 検索はフォルダ横断で「特定の1通を探す」行為のため束ねない。
- **誤爆耐性**: 束ねは件名ではなく **References/threadId ベース**（§1.5）なので、Gmailで嫌われる「同じ件名なだけの別件が束なる」事故は構造的に起きにくい。

**既知の制約**
- 集約は**取得済みウィンドウ内**（各フォルダ直近50通）で行う。会話の他メンバーが50通から外れた場合、その行はバッジなし（1通）に見える。全件正確な件数にはプロバイダ側の `threads.list` 相当が必要（将来改善）。
- リーダー（開いた画面）のアーカイブは選択中の1通のみに作用（会話単位にするかは要検討）。

## 2. インボックスの整理（バンドル／ビュー）

「送信者で散らばって見づらい」を解く。鍵は **AIによる自動カテゴリ分け** ＋ **束ねて一望できるUI**。

### 2.1 カテゴリ（重要度とは別軸）
[02](02-importance-triage-learning.md) の判定に、**種別（category）**を追加する:

| category | 例 | 既定の扱い |
|----------|----|-----------|
| `person` | 人からの個別メール | 最優先・展開 |
| `notification` | GitHub等の自動通知 | 束ねる |
| `newsletter` | 配信・メルマガ | 束ねる・既読化しやすく |
| `receipt` | 請求・領収・予約確認 | 束ねる（後で参照） |
| `calendar` | 招待・日程 | 束ねる＋日付で |
| `promotion` | 宣伝 | 束ねる・低 |

- category も**学習対象**（[02](02-importance-triage-learning.md) と同じ機構: シグナル＋RAG＋嗜好プロファイル）。送信者/ドメインで安定するので層0で大半が即断できる。

### 2.2 整理ビュー（3つの見せ方を切替）
1. **バンドル表示（既定）**: カテゴリで束ねて折りたたみ。
   ```
   ▸ 👤 人から (3)            ← 展開して個別に見る
   ▸ 🔔 通知 (12)            ← まとめて既読/アーカイブ
   ▸ 🧾 請求・領収 (2)
   ▸ 📰 ニュースレター (8)    ← 一括でさばける
   ```
   束ねた単位で**「全部アーカイブ」「全部既読」**＝消し込みが一気に進む。
2. **送信者レーン表示**: 送信者/ドメインごとにグルーピング。ノイズの多い送信者を畳んで、人を上に。
3. **スマートビュー（保存フィルタ）**: 「重要のみ」「人から」「未読」「今日」など。サイドバーに常駐。

### 2.3 時間グルーピング
どのビューでも一覧は **今日 / 今週 / それ以前** のセクションに（走査しやすさ）。

### 2.4 「澄ます」操作（消し込みの加速）
- バンドル右の **一括操作**（全アーカイブ/全既読/全ゴミ箱）。
- 「ニュースレターは常に束ねて自動既読」のような**ルール化**（カテゴリ学習と連動）。
- 目標: 朝、`person`/`重要` だけ数件残し、残りは束ねて一掃 → **受信箱が澄む**。

### 2.5 スター付きビュー（2026-06-12 実装）
カテゴリバンドルに先行して、**フォルダ横断のスター（お気に入り）**を実装した。
- サーバ正本: Gmail = `STARRED` ラベル、IMAP = `\Flagged` フラグ（他のメーラーと相互に見える）。provider seam の `setStarred()` で読み書き。
- `FolderView = MailboxState | "starred"` — スターはフォルダではなく**フラグ**なので、メールの `state` とは別軸（型でも区別）。
- スター付きビューはローカルキャッシュから提供（各フォルダの live fetch で星状態も write-through される）。キャッシュ外の古いメールは出ない制約あり。
- 暗黙学習との接続: [02](02-importance-triage-learning.md) §5.3 の「スターを付けた＝重要」シグナルの入力源になる（接続は未実装）。
