# 設計書 01 — 逐次的な返信添削UX（一箇所ずつ提案 → 採用/却下）

最終更新: 2026-06-03

## 1. 解決する課題

**現状（Phase 1 MVP）**: 「もっと丁寧に」等の指示を出すたびに、AIが**下書き全文を丸ごと差し替える**。
**問題**: どこがどう変わったか分からない／意図しない箇所まで変わる／ユーザーが主導権を持てない。

**ありたい姿**:
> 全体の下書きはまず出る。そのうえで「**ここをこうしてほしい**」と部分ごとに指示し、AIが**その箇所だけの修正案**を出す。ユーザーは**採用/却下を一つずつ**選びながら、少しずつ仕上げる。

これは Google Docs の「提案モード」や Cursor のインライン編集に近い、**提案（suggestion）駆動の編集**。

## 2. インタラクションモデル

3つの入口を用意し、すべて「提案 → 採用/却下」に収束させる。

```
① 範囲選択して指示        ② 全体への指示              ③ クイック操作
  本文の一部をドラッグ      「結びを柔らかく」          「丁寧に」「短く」等
  → フローティング         → 複数箇所の提案が          → ②と同じ（全体だが
     ツールバー「AIで修正」    まとめて出る                 提案として出る）
        │                        │                          │
        └────────────┬───────────┴──────────────────────────┘
                     ▼
        AIが「提案（suggestion）」を生成
        各提案 = { 対象スパン, 置換後, 理由 }
                     ▼
        本文中にインライン差分で表示
        （旧=取り消し線, 新=緑のハイライト）
                     ▼
        ユーザーが提案ごとに [採用 ✓] / [却下 ✗]
        （Tab=次の提案へ / ⌘↵=全採用 も可）
```

### 2.1 重要な設計判断

- **AIは本文を直接書き換えない**。常に「保留中の提案」として積み、確定はユーザーの採用操作のみ。
- **提案は複数同時に保留可能**。1回の指示で複数箇所の提案が出ても、個別に採否できる。
- **会話履歴は残す**（右レール）。各ターンは「本文を更新」ではなく「N件の提案を作成」になる。
- 手書き編集はいつでも可能。手書きと提案は共存する。

## 3. データモデル

```ts
// 1つの修正提案
interface Suggestion {
  id: string;
  /** 置換対象の元テキスト（アンカー）。本文中から一意に特定するために使う。 */
  anchor: string;
  /** anchor の本文内オフセット（生成時点）。ズレ検出に使う。 */
  anchorStart: number;
  anchorEnd: number;
  /** 置換後テキスト（削除提案なら空文字）。 */
  replacement: string;
  /** なぜこの修正をしたか（1行）。 */
  reason: string;
  status: "pending" | "accepted" | "rejected";
}

interface RefineTurn {
  id: string;
  instruction: string;            // ユーザーの指示
  scope: "selection" | "whole";   // 範囲指定か全体か
  suggestions: Suggestion[];
}
```

### アンカリング（ズレ対策）
AIが返す `anchor`（元テキスト断片）を、適用時に本文から検索して位置を再特定する。

1. `anchorStart/End` 付近で完全一致を試す。
2. 失敗時は本文全体で `anchor` 文字列検索（最初の一致）。
3. それも失敗（本文が変わった）なら、その提案を **`stale`** として「再生成」ボタンを出す。

> 提案を**オフセットではなくテキスト断片で持つ**ことで、他の提案を採用して本文がズレても適用できる。

## 4. API設計

### `POST /api/ai/suggest`

```ts
// Request
{
  email: Email;                    // 返信対象（文脈）
  draft: string;                   // 現在の下書き全文
  instruction: string;             // 「ここを丁寧に」等
  selection?: { start: number; end: number; text: string }; // 範囲指定時
}

// Response（generateObject で構造化出力）
{
  suggestions: Array<{
    anchor: string;        // draft 内に実在する部分文字列
    replacement: string;
    reason: string;
  }>;
}
```

**プロンプト方針**（システム）:
- 「下書き**全体は書き直さない**。指示に関係する箇所だけ、最小の差分で提案する」
- 「各提案の `anchor` は下書きに**実在する連続した部分文字列**であること」
- `selection` がある場合は「その範囲のみ」を対象にする。
- 構造化出力（`zod` スキーマ）で `Suggestion[]` を強制 → パース不要。

### ストリーミング
提案は `streamObject`（AI SDK v6）で**生成順に1件ずつ届ける**ことが可能。届いた提案から順にインライン表示すると「少しずつ出てくる」体験になる。

## 5. エディタ技術の選定

「インライン差分＋採用/却下」を素の `<textarea>` で実現するのは困難（装飾が乗らない）。リッチテキスト編集基盤が必要。

| 候補 | 長所 | 短所 | 判定 |
|------|------|------|------|
| **Tiptap (ProseMirror)** | 提案/トラックチェンジ向けの「mark」機構。装飾・採否・コメントが自然に作れる。Reactバインディング有り | 学習コスト中・バンドル増 | **推奨** |
| CodeMirror 6 | デコレーション（diff装飾）が得意・軽量 | プレーンテキスト寄りで「提案mark」は自作 | 次点 |
| 自作オーバーレイ | 依存なし | diff装飾・選択・IME（日本語入力）対応を自前で作るのは高コスト・バグ源 | 不可 |
| `<textarea>`+横並びdiff | 実装が一番軽い | インライン採否ができず「一箇所ずつ」体験にならない | Phase1暫定のみ |

**推奨**: **Tiptap**。提案を ProseMirror の inline mark（`suggestion`）として表現し、各 mark に採用/却下ボタンを付ける。日本語IME・装飾・部分選択すべてが実績ある基盤に乗る。

> 段階移行案: Phase 1.5 として、まず `diff-match-patch` による**ハンク単位の差分プレビュー＋採否**（textareaベース）で「提案→採否」体験を先に出し、Tiptap導入はその後でも良い。コストと体験のトレードオフ。

## 6. コンポーネント構成

```
ReplyComposer
├── DraftEditor (Tiptap)         本文＋保留中提案のインライン表示
│   └── SuggestionMark           1提案ぶんの装飾＋[✓][✗]
├── SelectionToolbar             範囲選択時に浮かぶ「AIで修正」＋クイック
├── AssistantRail (右)
│   ├── TurnList                 会話履歴（各ターン＝N件の提案）
│   ├── QuickPrompts             丁寧に/短く/英語に…（全体スコープ）
│   └── InstructionInput         自由指示
└── SendBar                      今すぐ送信 / 予約送信 / 破棄
```

### キーボード
- `Tab` / `Shift+Tab`: 次/前の保留提案へフォーカス
- `⌘↵`: フォーカス中の提案を採用（連続採用）
- `⌘⌫`: 却下
- `⌘⇧↵`: 全提案を採用

## 7. 受け入れ基準（Phase 1.5 / 2）

- [ ] 範囲選択 → 指示 → その範囲**だけ**の提案が出る
- [ ] 全体指示で複数箇所の提案が出て、**個別に**採否できる
- [ ] 採用すると本文に反映、却下すると消える。未採用の提案は本文を変えない
- [ ] 本文が変わっても残りの提案が（アンカー再特定で）適用できる／無理なら stale 表示
- [ ] 提案ごとに理由が見える
- [ ] 日本語IME入力中に装飾が壊れない

## 8. 現行実装との関係

現状の `POST /api/ai/chat`（全文差し替え）は **Phase 1 の暫定**。本設計の `POST /api/ai/suggest` がそれを置き換える。クイック操作（丁寧に等）も、全文置換ではなく「全体スコープの提案生成」に変更する。

## 9. 実装メモ（2026-06-12 追記 — 件名の自動提案）

`POST /api/ai/suggest` に**件名提案**を追加した（「件名を考えるのがしんどい」対応）:

- リクエストに現在の `subject` を同送。**未入力（空）の場合のみ**、本文修正と同時に件名（日本語・25文字以内・Re:/Fwd: なし）を `subject` として返す（スキーマを動的に切替）。
- クライアント（ReplyComposer）は、提案が返った時点でも**まだ件名が空のままなら**自動入力し「件名も提案しました（変更できます）」と表示。ユーザーが入力済みの件名は決して上書きしない。
- AIで返信（`/api/ai/reply`）は従来から subject+body を返すため対象外。本機能は新規メール・転送など件名が空のまま添削に入るケース向け。
